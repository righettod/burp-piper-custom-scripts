    <article class="markdown-body entry-content container-lg" itemprop="text"><h1><a id="user-content-openid-client" class="anchor" aria-hidden="true" href="#openid-client"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>openid-client</h1>
<p>openid-client is a server side <a href="https://openid.net/connect/" rel="nofollow">OpenID</a> Relying Party (RP, Client) implementation for
Node.js runtime, supports <a href="http://passportjs.org" rel="nofollow">passport</a>.</p>
<h2><a id="user-content-implemented-specs--features" class="anchor" aria-hidden="true" href="#implemented-specs--features"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Implemented specs &amp; features</h2>
<p>The following client/RP features from OpenID Connect/OAuth2.0 specifications are implemented by
openid-client.</p>
<ul>
<li><a href="https://openid.net/specs/openid-connect-core-1_0.html" rel="nofollow">OpenID Connect Core 1.0</a>
<ul>
<li>Authorization Callback
<ul>
<li>Authorization Code Flow</li>
<li>Implicit Flow</li>
<li>Hybrid Flow</li>
</ul>
</li>
<li>UserInfo Request</li>
<li>Fetching Distributed Claims</li>
<li>Unpacking Aggregated Claims</li>
<li>Offline Access / Refresh Token Grant</li>
<li>Client Credentials Grant</li>
<li>Client Authentication
<ul>
<li>none</li>
<li>client_secret_basic</li>
<li>client_secret_post</li>
<li>client_secret_jwt</li>
<li>private_key_jwt</li>
</ul>
</li>
<li>Consuming Self-Issued OpenID Provider ID Token response</li>
</ul>
</li>
<li><a href="https://tools.ietf.org/html/rfc8414" rel="nofollow">RFC8414 - OAuth 2.0 Authorization Server Metadata</a> and <a href="https://openid.net/specs/openid-connect-discovery-1_0.html" rel="nofollow">OpenID Connect Discovery 1.0</a>
<ul>
<li>Discovery of OpenID Provider (Issuer) Metadata</li>
<li>Discovery of OpenID Provider (Issuer) Metadata via user provided inputs (via <a href="https://github.com/panva/node-openid-client/blob/master/docs/README.md#issuerwebfingerinput">webfinger</a>)</li>
</ul>
</li>
<li><a href="https://openid.net/specs/openid-connect-registration-1_0.html" rel="nofollow">OpenID Connect Dynamic Client Registration 1.0</a>
<ul>
<li>Dynamic Client Registration request</li>
<li>Client initialization via registration client uri</li>
</ul>
</li>
<li><a href="https://tools.ietf.org/html/rfc7009" rel="nofollow">RFC7009 - OAuth 2.0 Token revocation</a>
<ul>
<li>Client Authenticated request to token revocation</li>
</ul>
</li>
<li><a href="https://tools.ietf.org/html/rfc7662" rel="nofollow">RFC7662 - OAuth 2.0 Token introspection</a>
<ul>
<li>Client Authenticated request to token introspection</li>
</ul>
</li>
<li><a href="https://tools.ietf.org/html/rfc8628" rel="nofollow">RFC8628 - OAuth 2.0 Device Authorization Grant (Device Flow)</a></li>
<li><a href="https://tools.ietf.org/html/rfc8705" rel="nofollow">RFC8705 - OAuth 2.0 Mutual TLS Client Authentication and Certificate-Bound Access Tokens</a>
<ul>
<li>Mutual TLS Client Certificate-Bound Access Tokens</li>
<li>Metadata for Mutual TLS Endpoint Aliases</li>
<li>Client Authentication

<li>self_signed_tls_client_auth</li>
</ul>
</li>
</ul>
</li>
<li><a href="https://openid.net/specs/openid-connect-session-1_0.html#RPLogout" rel="nofollow">OpenID Connect Session Management 1.0 - draft 28</a>
<ul>

</ul>
</li>
<li><a href="https://openid.net/specs/openid-financial-api-part-2-ID2.html" rel="nofollow">Financial-grade API - Part 2: Read and Write API Security Profile (FAPI) - ID2</a></li>
<li><a href="https://openid.net/specs/openid-financial-api-jarm-ID1.html" rel="nofollow">JWT Secured Authorization Response Mode for OAuth 2.0 (JARM) - ID1</a></li>
<li><a href="https://tools.ietf.org/html/draft-ietf-oauth-dpop-01" rel="nofollow">OAuth 2.0 Demonstration of Proof-of-Possession at the Application Layer (DPoP) - draft 01</a></li>
<li><a href="https://tools.ietf.org/html/draft-ietf-oauth-par-06" rel="nofollow">OAuth 2.0 Pushed Authorization Requests (PAR) - draft 06</a></li>
</ul>
<p>Updates to draft specifications (DPoP, JARM, and FAPI) are released as MINOR library versions,
if you utilize these specification implementations consider using the tilde <code>~</code> operator in your

OpenID Connect Authorization Server responses or wrappers around requests to its endpoints. Aside
from a generic OpenID Connect <a href="http://passportjs.org" rel="nofollow">passport</a> strategy it does not expose neither express

<p>Authorization Code flow is for obtaining Access Tokens (and optionally Refresh Tokens) to use with
third party APIs securely as well as Refresh Tokens. In this quick start your application also uses
PKCE instead of <code>state</code> parameter for CSRF protection.</p>
<p>Create a Client instance for that issuer's authorization server intended for Authorization Code
flow.</p>
<p><strong>See the <a href="https://github.com/panva/node-openid-client/blob/master/docs/README.md">documentation</a> for full API details.</strong></p>

<p>Implicit <code>response_type=id_token</code> flow is perfect for simply authenticating your end-users, assuming
the only job you want done is authenticating the user and then relying on your own session mechanism
with no need for accessing any third party APIs with an Access Token from the Authorization Server.</p>
<p>Create a Client instance for that issuer's authorization server intended for ID Token implicit flow.</p>
<p><strong>See the <a href="https://github.com/panva/node-openid-client/blob/master/docs/README.md">documentation</a> for full API details.</strong></p>

<p><strong>Yes.</strong> Everything that's either exported in the TypeScript definitions file or
<a href="https://github.com/panva/node-openid-client/blob/master/docs/README.md">documented</a> is subject to
<a href="https://semver.org/spec/v2.0.0.html" rel="nofollow">Semantic Versioning 2.0.0</a>. The rest is to be considered
private API and is subject to change between any versions.</p>
